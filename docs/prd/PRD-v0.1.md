# 北京记忆地图 PRD v0.1

## 1. 产品定位

北京记忆地图是一款本地优先的多端独立客户端，面向刚到北京工作和生活的人。产品以互动地图为核心，把通勤理解、城市认知、周末探索、个人打卡记忆和 AI 个性化助手结合起来，用“解锁地图”的方式帮助用户快速、有趣地熟悉北京。

当前个人化起点：用户在字节跳动大钟寺工区工作，第一视角从大钟寺、知春路、中关村、五道口、西直门、学院路等北三环/海淀区域开始，但产品范围覆盖北京全域。

一句话概念：以你的通勤和足迹为线索，逐步解锁属于你的北京。

## 2. 目标用户

### 2.1 核心用户

- 刚来北京工作的新人
- 希望快速建立北京空间感的人
- 需要理解通勤、生活圈、社交探索和周末路线的人
- 希望把自己在北京的经历沉淀成地图记忆的人

### 2.2 当前用户画像

- 工作地点：字节跳动大钟寺工区
- 关注点：通勤、城市认知、社交探索、周末了解整个北京版图
- 偏好：游戏地图/手绘风格、有特色、可解锁、可记录
- 数据策略：全部本地存储
- AI：允许配置自己的 AI key

## 3. 产品目标

### 3.1 第一阶段目标

- 让用户在地图上理解北京的基础结构：环路、行政区、地铁骨架、核心生活圈。
- 让用户围绕大钟寺建立工作日通勤和下班活动半径。
- 让用户通过周末路线逐步了解北京全域。
- 让用户可以手动打卡地点，上传图片和文字，形成个人城市记忆。
- 让用户可以配置 AI key，由 AI 基于本地足迹和兴趣生成路线、解释区域、总结回忆。

### 3.2 非目标

- 第一版不做实时导航替代高德/百度地图。
- 第一版不做社交公开社区。
- 第一版不做云同步。
- 第一版不依赖线上账号体系。
- 第一版不做复杂商业 POI 排名。

## 4. 平台范围

### 4.1 第一版

- 桌面端优先：Windows/macOS 独立客户端。
- 手机端先做响应式设计和架构预留，不在 MVP 强行完成原生移动端。

### 4.2 后续

- 移动端：React Native、Flutter 或 Capacitor 方案待技术验证后确定。
- 数据同步：默认本地，后续可选 WebDAV、iCloud、用户自建对象存储或端到端加密同步。

## 5. 核心功能

### 5.1 互动地图首页

地图是主界面，用户打开应用后直接进入北京地图，而不是信息流或营销页。

地图图层：

- 行政区图层：北京市 16 区。
- 环路图层：二环、三环、四环、五环、六环认知。
- 地铁图层：重点展示与大钟寺相关的 10 号线、13 号线、4 号线、昌平线、2 号线等，以及北京整体地铁骨架。
- 生活圈图层：中关村、五道口、西直门、三里屯、国贸、望京、798、亮马河、前门、什刹海、亦庄、通州等。
- 我的足迹图层：导入行程和手动打卡形成的个人轨迹、点位和热力。
- 待探索图层：AI 推荐或用户计划的路线和地点。

### 5.2 游戏化/手绘地图风格

视觉方向：

- 偏游戏地图和手绘认知地图，不做普通导航地图的冷冰冰风格。
- 区域可带有插画化符号，例如中轴线、长城、大学、CBD、高铁站、公园、胡同、机场等。
- 未解锁区域半透明或雾化；已学习、已去过、已打卡区域逐步点亮。
- 地图标记应有记忆感：照片打卡点、文字笔记点、路线计划点、通勤锚点有不同视觉状态。

设计要求：

- 游戏感服务于城市认知，不做低幼化。
- 保留基础方位和地理准确性，不能让用户失去真实空间感。
- 支持真实地图与手绘地图之间切换，第一版可先使用真实底图叠加手绘风格覆盖层。

### 5.3 大钟寺工作锚点

默认工作锚点为字节跳动大钟寺工区。围绕该锚点提供：

- 工作日 30/45/60 分钟活动圈。
- 到核心城区和商圈的大致空间理解。
- 常见通勤方向分析。
- 下班后可探索地点推荐。
- 周末从大钟寺出发的城市路线建议。

重点关联区域：

- 大钟寺
- 知春路
- 西土城
- 五道口
- 中关村
- 海淀黄庄
- 学院路
- 西直门
- 动物园
- 什刹海/鼓楼方向

### 5.4 北京全域城市认知

内置北京城市知识库，帮助用户理解全北京版图。

行政区认知：

- 东城/西城：老北京、中轴线、胡同、核心历史文化。
- 海淀：高校、科研、互联网、知识密度。
- 朝阳：CBD、使馆区、三里屯、亮马河、望京、社交和国际化。
- 丰台：交通枢纽、丽泽、南城更新。
- 石景山：首钢、山水、冬奥遗产。
- 通州：城市副中心、大运河。
- 昌平：北部居住带、回龙观、生命科学园、十三陵。
- 顺义：机场、会展、国际学校、别墅区。
- 大兴/亦庄：新机场、产业园区、南部新城。
- 门头沟/房山/怀柔/密云/延庆/平谷：山水、长城、远郊周末探索。

城市结构认知：

- 环路如何影响北京人的位置感。
- 北部通勤带、东部社交带、西部高校文化、南城更新、远郊自然线。
- 地铁如何构成北京新人的城市骨架。

### 5.5 周末探索与计划路线

用户可以创建计划路线，作为“待探索任务”。

路线类型：

- 市区文化：中轴线、故宫/景山、什刹海/鼓楼、雍和宫/国子监、前门/天坛。
- 社交生活：三里屯、亮马河、国贸、798、望京。
- 海淀知识线：中关村、五道口、北大清华、圆明园、颐和园。
- 公园自然：奥森、香山、玉渊潭、朝阳公园、温榆河。
- 远郊周末：慕田峪、古北水镇、百里画廊、潭柘寺、爨底下、十渡。

路线功能：

- 用户手动创建路线：选择地点、排序、备注、计划日期。
- AI 生成路线：根据起点、时间、兴趣、已去过地点、天气/季节偏好生成。
- 路线状态：待探索、已完成、部分完成、想再去。
- 完成后沉淀为记忆路线，可上传照片和文字。

### 5.6 地点打卡与城市记忆

每个地点可以创建记忆卡片。

字段：

- 地点名称
- 经纬度/地图位置
- 地址或区域
- 到访时间
- 图片
- 文字笔记
- 标签：美食、咖啡、夜景、公园、展览、朋友、工作、通勤、独处、第一次等
- 心情，可选
- 同行人，可选
- 消费，可选
- 交通方式，可选
- 是否加入某条路线
- 是否纳入 AI 总结

地图状态：

- 仅路过：浅色点
- 手动打卡：实心点
- 有图片：图片缩略标记
- 有文字：笔记标记
- 多次到访：时间线聚合

记忆能力：

- 按区域查看打卡。
- 按时间线查看打卡。
- 搜索笔记和标签。
- 生成“我的北京月报”。
- 回答“我上次在哪里拍过夜景？”这类基于本地记忆的问题。

### 5.7 高德行程记录导入

目标：把用户已有高德行程转化为个人足迹。

第一版支持：

- CSV/JSON/KML/GPX 文件导入中的一种或多种，具体取决于高德可导出的实际数据形态。
- 如果高德无法直接导出，预留截图识别、手动导入、位置文件导入等替代路径。

导入后能力：

- 生成足迹点和轨迹。
- 统计常去区域、常走线路、活动半径。
- 自动点亮去过的区域。
- 和手动打卡合并展示。

待验证问题：

- 高德 App 当前是否支持导出行程记录。
- 可导出字段包含哪些：时间、经纬度、地点名、交通方式、路线轨迹等。
- 数据格式是否稳定。

### 5.8 AI 个性化助手

用户自行配置 AI key，应用只调用用户设置的模型服务。

配置项：

- API base URL
- API key
- model
- 是否允许读取本地足迹
- 是否允许读取打卡笔记
- 是否允许读取图片描述

AI 能力：

- 解释北京区域和城市结构。
- 根据大钟寺出发推荐下班路线和周末路线。
- 根据已解锁地图推荐下一处探索。
- 总结个人足迹和记忆。
- 根据打卡笔记生成月报、年度回顾。
- 把自然语言计划转成地图路线。

隐私原则：

- AI key 只保存在本地。
- 默认不上传完整个人数据，用户需要明确授权。
- 每次 AI 请求前展示或记录将发送的数据范围。
- 后续需要做本地数据脱敏和最小上下文发送。

## 6. 解锁系统

解锁方式：

- 去过：由高德行程或手动打卡解锁。
- 学过：阅读区域认知卡片后解锁。
- 计划过：加入待探索路线后进入半解锁状态。
- 深度探索：上传图片和文字后提升探索等级。

解锁对象：

- 行政区
- 生活圈
- 地铁线
- 环路区域
- 经典路线
- 城市主题

示例成就：

- 第一次理解三环
- 从大钟寺出发的第一次周末探索
- 点亮海淀知识线
- 穿越北京中轴线
- 第一次远郊周末
- 第一次记录一条完整路线

## 7. 数据策略

默认本地优先。

本地数据：

- 用户设置
- AI key
- 公司/住处/常用地点
- 高德导入数据
- 打卡图片和文字
- 计划路线
- 解锁进度
- AI 生成的总结和推荐

建议技术：

- SQLite 存结构化数据。
- 本地文件系统存图片原图和缩略图。
- 数据库只保存图片路径、hash、元数据。
- 敏感配置使用系统安全存储能力，Electron 可考虑 keytar 或系统凭据管理。

备份策略：

- 第一版支持手动导出备份包。
- 备份包包含 SQLite、图片目录、配置元数据。
- AI key 默认不进入备份包，除非用户明确选择。

## 8. MVP 范围

### 8.1 必须做

- 桌面端项目骨架。
- 北京互动地图主界面。
- 手绘/游戏化视觉方向验证。
- 工作锚点：大钟寺。
- 北京行政区、环路、核心生活圈基础知识库。
- 手动地点打卡。
- 图片和文字记录。
- 计划路线创建和状态管理。
- 本地 SQLite 存储。
- AI key 配置。
- AI 基于本地数据生成探索建议。
- 基础解锁进度。

### 8.2 可选做

- 高德行程导入第一版解析器。
- 路线时间线回放。
- 月度记忆总结。
- 地图截图分享。

### 8.3 暂不做

- 云同步。
- 社区分享。
- 实时导航。
- 商业推荐排序。
- 多人协作路线。

## 9. 推荐技术栈

第一版：

- 桌面客户端：Electron
- 前端：React + TypeScript
- 构建：Vite
- 地图：高德地图 Web SDK
- 本地数据库：SQLite
- 本地图片：文件系统 + 缩略图生成
- AI：OpenAI-compatible API 配置
- 状态管理：Zustand 或 Redux Toolkit，优先轻量方案
- UI：自定义地图游戏化 UI，组件库只用于基础控件

后续：

- 移动端：React Native / Flutter / Capacitor 待验证
- 共享包：core、data、ai、map-style
- 同步：端到端加密备份或用户自选同步服务

## 11. 关键风险

- 高德行程记录导出能力不确定，需要优先调研。
- 高德地图 SDK 在 Electron 中的 key、域名、安全限制需要验证。
- 手绘地图与真实地图的叠加需要控制地理准确性。
- AI 读取个人足迹和照片描述存在隐私风险，需要明确授权和最小化发送。
- 多端同步会显著增加复杂度，MVP 不建议做。

## 12. 下一步建议

1. 调研高德行程记录可导出格式。
2. 做 Electron + React + 高德地图最小技术验证。
3. 建立北京基础知识库的数据结构。
4. 设计手绘/游戏化地图视觉稿。
5. 建立本地 SQLite schema。
6. 做地点打卡和计划路线的核心交互原型。
