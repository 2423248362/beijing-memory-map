# 北京记忆地图 PRD v0.3

## 1. 版本定位

v0.3 的目标不是继续扩展「地图 + 打卡 + AI + 路线」的功能清单，而是验证一个更小、更清晰的产品假设：

> 刚来北京的人，如果每天从自己的工作锚点出发完成一个轻量城市认知任务，就能更快建立可用的北京方向感，并愿意把真实探索记录回地图。

本版本采用 Lean UX 思路：先明确假设和行为指标，再决定功能。所有功能都必须服务一个核心闭环：

1. 用户知道今天该理解北京的哪一小块。
2. 用户能在地图上看懂这个地点和自己的关系。
3. 用户能把它变成一条可执行路线或一次记忆记录。
4. 系统能根据用户行为展示「我已经理解了哪些方向，还缺什么」。

v0.3 的设计评分目标为 Lean UX 8/10：减少冗余功能，明确假设与验证方式，但仍保留部分未验证的 AI 和地图能力作为实验入口。达到 10/10 需要真实用户测试数据和实验日志。

## 2. 核心用户与使用场景

### 2.1 核心用户

- 刚来北京工作 0-6 个月的人。
- 工作地点相对固定，默认锚点为「字节跳动大钟寺工区」。
- 知道一些地名，但缺少空间关系、通勤方向和生活场景判断。
- 不想再收藏大量攻略，更需要「今天下班先去哪」「周末从哪里开始」的具体建议。

### 2.2 主要场景

- **下班前 5 分钟**：打开应用，想知道今天可以理解哪个近处地点。
- **周末前一晚**：想选一条半日路线，但不想自己查攻略和串点。
- **探索结束后**：想记录去过哪里、为什么记住这里，而不是只存照片。
- **一段时间后**：想知道自己熟悉了北京哪些方向，还缺哪些区域。

## 3. Lean UX 假设

### 3.1 高风险假设

| ID | 假设 | 风险 | 不确定性 | 处理 |
| --- | --- | --- | --- | --- |
| H1 | 新人愿意以「工作锚点」作为理解北京的起点 | 高 | 中 | v0.3 重点验证 |
| H2 | 「今日认知任务」比展示大量地点更能推动行动 | 高 | 高 | v0.3 重点验证 |
| H3 | 用户愿意在探索后写一条简短记忆 | 中 | 高 | v0.3 观察验证 |
| H4 | AI 城市建议能提高路线创建率 | 中 | 高 | 作为实验入口，不作为核心依赖 |
| H5 | 真实高德地图是必要底座 | 高 | 低 | 保留并优化配置引导 |

### 3.2 核心产品假设

我们相信「新用户首日完成率」会提升，如果刚来北京工作的用户能在首次打开应用时，通过「今日认知」看到一个从大钟寺出发的具体任务，并在 3 分钟内完成「查看地点档案」或「创建路线」。

成功信号：

- 80% 的测试用户能说出当前推荐任务是什么。
- 70% 的测试用户能在 3 分钟内完成地点档案查看。
- 50% 的测试用户会创建一条路线或进入记忆记录。

### 3.3 子假设

- H2-A：用户更容易理解「大钟寺 -> 知春路」这类短线任务，而不是全北京地图点位。
- H2-B：地点卡中「为什么新人需要知道这里」比百科信息更有用。
- H2-C：进度面板如果按方向和区域表达，会比单纯统计数量更能激励继续探索。

## 4. 产品原则

### 4.1 结果优先，不以功能数量证明价值

v0.3 不追求功能完整，而追求用户行为改变。功能是否保留取决于它是否帮助用户完成核心闭环。

### 4.2 地图是空间底座，任务是行动入口

用户不应从一张满是点位的地图开始，而应从一个可执行问题开始：

- 今天我该理解哪一块？
- 这地方离我的锚点是什么关系？
- 我应该走哪条线？
- 去完之后我记住了什么？

### 4.3 地点卡不是百科卡

地点卡只回答新人最需要的问题：

- 为什么新人需要知道这里？
- 它和我的工作锚点有什么关系？
- 适合什么时候去？
- 第一次去应该观察什么？
- 下一步行动是什么？

### 4.4 AI 是辅助实验，不是核心流程

AI 建议可以作为增强体验，但 v0.3 不依赖 AI 才能完成主流程。无 AI Key 时，用户仍应能完成今日任务、地点理解、路线创建和记忆记录。

## 5. v0.3 MVP 范围

### 5.1 必须保留

- 真实地图加载与无 Key 配置引导。
- 内置生活圈 marker 展示与点击。
- 右侧「今日 / 地点 / 路线 / 记忆 / 进度 / 设置」结构。
- 今日认知任务。
- 新人地点档案。
- 主题路线模板与路线创建。
- 地图点击生成记忆草稿。
- 本地打卡记忆保存。
- 基础进度面板。

### 5.2 必须弱化或删除

- 大面积 AI 入口，不作为首屏主任务。
- 装饰性导游、人物、动画和无行动价值的视觉元素。
- 过多并列指标，只保留生活圈、记忆、路线、完成率。
- 百科式地点描述。
- 无真实路径能力前，不宣称「真实导航」或「完整路线规划」。
- 未验证前，不做复杂成就系统、社交、云同步、移动端。

### 5.3 暂不做

- 商业 POI 排名。
- 实时导航。
- 多人协作和分享。
- 图片管理详情页。
- 大规模内容平台。
- 自动生成完整 30 天报告。

## 6. 信息架构

### 6.1 主界面

- **顶栏**：产品名、全局搜索、工作锚点、地图设置。
- **地图区**：真实地图、地点 marker、路线 polyline、关键统计浮层。
- **右侧面板**：任务和解释的主工作区。

### 6.2 右侧面板

| 面板 | 目标 | 必须回答的问题 |
| --- | --- | --- |
| 今日 | 给用户一个明确下一步 | 今天先理解什么？为什么是它？ |
| 地点 | 建立地点认知 | 这个地方为什么对新人重要？ |
| 路线 | 把认知变成行动 | 从大钟寺出发怎么走一条学习路径？ |
| 记忆 | 沉淀个人地图 | 我去过哪里，留下了什么判断？ |
| 进度 | 展示熟悉与空白 | 我已经熟悉哪些方向，还缺什么？ |
| 设置 | 保证地图可用 | 高德 Key 是否配置成功？ |

## 7. 核心用户流程

### 7.1 首次打开

1. 用户进入应用。
2. 如果未配置高德 Key，地图区显示配置引导，右侧仍显示今日认知。
3. 用户看到默认锚点「大钟寺」和首个任务。
4. 用户点击「地点档案」，理解大钟寺和北三环/知春路/西直门的关系。
5. 用户选择「规划路线」或「记录记忆」。

验收标准：

- 用户不需要读说明书即可知道第一步。
- 无地图 Key 时也不出现死路。

### 7.2 完成第一条短线

1. 用户在今日面板看到「大钟寺到知春路」任务。
2. 用户打开路线面板，使用工作日短线模板。
3. 地图显示路线 polyline。
4. 用户点击路线停靠点查看地点档案。
5. 探索后在记忆面板保存一次打卡。

验收标准：

- 用户能复述这条路线的认知目标，而不只是知道地点名。

### 7.3 发现一个想去的地点

1. 用户搜索「亮马河」或点击地图 marker。
2. 系统打开地点档案。
3. 用户看到它和朝阳社交线、水岸生活的关系。
4. 用户创建路线或记录为计划探索。

验收标准：

- 搜索结果不只是跳到地图点，而是进入可行动的地点解释。

### 7.4 查看进度

1. 用户打开进度面板。
2. 系统展示已打卡、已创建路线、已理解点位和未探索方向。
3. 用户选择一个空白方向继续探索。

验收标准：

- 进度不只是数字，而能指导下一步。

## 8. 功能需求

### 8.1 今日认知

必须展示：

- 当前选中地点。
- 一个匹配当前地点或默认阶段的今日任务。
- 推荐路线模板。
- 三个行动：生成城市建议、规划路线、查看地点档案、记录记忆。

规则：

- 如果当前地点匹配某个任务目标，优先显示该任务。
- 如果没有匹配任务，显示首个入门任务。
- AI 生成按钮可禁用，但不能阻断其他行动。

### 8.2 地点档案

必须展示：

- 地点名、行政区、认知状态。
- 认知意义。
- 与工作锚点关系。
- 适合时间。
- 适合同伴。
- 推荐行动。
- 关联城市概念。

禁止：

- 写成长百科。
- 用无关图标或装饰替代内容。

### 8.3 路线学习

必须支持：

- 从当前地点生成「大钟寺 -> 当前地点」路线。
- 使用内置路线模板。
- 展示已创建路线。
- 点击已创建路线后地图高亮对应 polyline。

限制：

- v0.3 只保证认知路线，不承诺真实导航路径。
- 高德真实路线规划作为后续实验，不阻塞 v0.3。

### 8.4 记忆记录

必须支持：

- 地图点击生成打卡草稿。
- 当前地点直接记录打卡。
- 输入笔记。
- 可选择图片。
- 保存后在地图上出现个人记忆 marker。

v0.3 不要求：

- 图片编辑。
- 记忆详情页。
- 社交分享。

### 8.5 进度面板

必须展示：

- 已打卡记忆数。
- 已创建路线数。
- 已完成路线数。
- 已理解点位数。
- 已熟悉区域。
- 建议补齐方向。

后续优化：

- 将「方向」从硬编码区域升级为真实城市方向模型。

### 8.6 AI 城市建议

必须满足：

- 没有 API Key 时清晰提示，并允许保存配置。
- 有 API Key 时基于当前地点生成结构化建议。
- AI 输出可以生成路线，但不能替代用户确认。

## 9. 数据与技术边界

### 9.1 复用现有数据结构

v0.3 不新增数据库表，优先复用：

- `LifeCircle`
- `CityTask`
- `RouteTemplate`
- `Checkin`
- `ExplorationRoute`
- `MapSettings`
- `AiSettings`

### 9.2 允许补充的前端派生逻辑

- 根据当前地点匹配今日任务。
- 根据当前地点匹配推荐路线。
- 根据打卡和路线计算基础进度。
- 根据地点状态生成 marker 样式。

### 9.3 不在 v0.3 中解决

- 完整 POI 搜索导入。
- 真实步行/公交路径规划。
- 多锚点管理。
- 用户账户系统。
- 云端同步。

## 10. 成功指标

### 10.1 行为指标

| 指标 | 目标 | 类型 |
| --- | --- | --- |
| 首屏第一步理解率 | 80% 测试用户能说出下一步 | Task Success |
| 地点档案完成率 | 70% 用户 3 分钟内打开地点档案 | Adoption |
| 首条路线创建率 | 50% 用户首次会话创建路线 | Trial |
| 首次记忆记录率 | 30% 用户首次或第二次会话保存打卡 | Trial |
| 任务目标复述率 | 60% 用户能说出路线认知目标 | Learning Outcome |

### 10.2 反指标

如果出现以下情况，说明产品仍然冗余或方向错误：

- 用户只缩放地图，不理解今日任务。
- 用户点击 AI，但不创建路线、不查看地点、不打卡。
- 用户认为这是攻略收藏工具，而不是城市认知工具。
- 用户无法解释「为什么从大钟寺开始」。

## 11. 实验计划

### 11.1 第一轮可用性测试

实验类型：中高保真可用性测试。

参与者：3-5 位刚来北京或不熟悉北京城区结构的人。

任务：

1. 找到今天推荐你理解的地点。
2. 说出这个地点和大钟寺的关系。
3. 创建一条从大钟寺出发的路线。
4. 记录一次你想留下的北京记忆。
5. 查看你还缺哪些探索方向。

记录：

- 是否完成。
- 完成时间。
- 卡在哪一步。
- 用户复述出的城市关系。
- 用户认为多余的功能。

### 11.2 第二轮内容测试

实验类型：内容理解测试。

目标：验证地点档案文案是否比百科式介绍更有帮助。

方法：

- 给同一地点两种文案：百科版、认知版。
- 让用户回答「我什么时候该去」「它和我的生活有什么关系」「下一步怎么做」。
- 比较回答准确率和信心评分。

## 12. 优先级

### P0：收敛核心闭环

- 今日认知首页。
- 地点档案。
- 路线模板创建。
- 记忆记录。
- 基础进度。
- 无 Key 地图引导。

### P1：让闭环更可信

- 地图点击后的草稿质量提升。
- 搜索结果更强地引导到地点档案。
- 路线创建后自动打开路线面板并高亮。
- 进度面板按方向而不只是按数量表达。

### P2：实验性增强

- AI 根据记忆生成下一条路线。
- POI 搜索导入。
- 高德真实路线规划。
- 30/60/90 分钟生活半径。

## 13. 验收标准

### 产品验收

- 新用户打开后能在 10 秒内知道今天该做什么。
- 点击任意核心地点后能看到新人认知卡。
- 用户能创建一条路线，并在地图上看到变化。
- 用户能保存一条记忆，并在地图上看到个人 marker。
- 进度面板能指出至少一个下一步探索方向。

### 技术验收

- `npm run typecheck` 通过。
- `npm run build` 通过。
- 无高德 Key 时不加载 SDK，并显示配置引导。
- 有高德 Key 时地图可加载、marker 可点击、路线可显示。
- 本地 SQLite 数据刷新后保留。
- 1280px、1440px、窄屏下无明显布局溢出。

## 14. 下一步实施建议

1. 将当前 UI 继续收敛为一条首屏闭环：今日任务 -> 地点档案 -> 路线/记忆。
2. 给 `CityTask` 和 `RouteTemplate` 补齐更强的阶段说明和行动文案。
3. 让路线创建后自动选中新路线，减少用户不确定感。
4. 改造进度面板，从数量统计升级为「方向缺口」。
5. 做 3-5 人可用性测试，把结果写入实验日志，再决定是否继续做 AI、POI 和真实路线规划。
